## 一、做得好的地方

### 1. Design-first 工作流

整个项目严格遵循 **需求 → 设计 → 实现** 的顺序。Phase 0 在写第一行代码之前完成了完整的文档体系：PRD、User Stories、User Flows、Wireframes、Database Design、API Design、Tech Architecture。这带来了几个明显的好处：

- **减少返工**：实现阶段几乎没有出现"做到一半发现设计有问题"的情况
- **追加需求有据可依**：功能 9（提醒通知）、功能 10-11（Auth + Landing Page）作为追加需求加入时，都先走完整的设计流程再动手，与主线功能保持了一致的质量标准
- **文档即沟通**：13 篇设计文档形成了清晰的决策链，任何阶段都能回溯"为什么这样做"

### 2. 设计决策记录（DD-001 ~ DD-015）

`design-decisions.md` 是这个项目最有价值的文档之一。每条决策都记录了 **背景、方案、备选方案、决策理由**，形成了可追溯的技术决策链。

亮点：
- DD-003（Streak 不持久化）避免了补卡场景下复杂的回溯更新逻辑
- DD-005（Milestone 快照）精准预判了"第 66 天触发 vs 第 80 天回看"的时间差问题
- DD-011（聚合接口 `/api/today`）在保持 RESTful 语义的同时解决了首屏性能问题
- DD-013（不引入 Agent SDK）体现了"够用就好"的工程判断力

### 3. 科学理念深度融入产品

不是简单地"做一个打卡工具"，而是将行为心理学研究有机融入每个功能设计：

| 科学原理 | 对应功能 | 融入方式 |
|----------|----------|----------|
| 习惯回路 | 创建习惯必选 Cue | 不是可选项，是产品约束 |
| 66 天法则 | 里程碑系统 | 5 级里程碑 + 数据快照 |
| 意志力有限 | 5 个习惯上限 | 硬性限制 + 科学解释 |
| 偶尔中断不致命 | 断裂安慰 + 补卡 | 温暖文案 + 月 3 次全局配额 |
| 习惯叠加 | Habit Stacking | 关联已有习惯 + 自动生成文案 |

这让项目从"CRUD demo"提升到了"有产品思考的全栈应用"。

### 4. 架构预留与渐进式扩展

- `authPlaceholder` 中间件从 Phase 1 就预留了 userId 透传，Phase 3.5 替换为真实 JWT 时 **Service 层零改动**
- `packages/shared` 实现了前后端类型和校验的 single source of truth（Zod schema → TypeScript type）
- Monorepo 结构清晰（shared/client/server），依赖关系单向，便于独立构建和测试

### 5. 务实的技术选型

每个技术选型都是"刚好够用"，没有过度工程化：
- SQLite（不是 PostgreSQL）— demo 项目不需要分布式数据库
- Web Notification API（不是 Web Push）— 不需要 Push Server 基础设施
- 本地 JWT（不是 OAuth）— clone 即用，评审者零配置
- Claude Haiku + fallback（不是 GPT-4）— 短文案生成，成本可忽略，API 不可用时优雅降级

### 6. 测试策略合理

80 个测试覆盖了三个层次：
- **Shared schemas**（28 tests）：校验规则的正确性
- **Server services**（31 tests）：核心业务逻辑（streak 计算、打卡、里程碑）
- **Server routes**（13 tests）：API 端到端集成（supertest）
- **Middleware**（8 tests）：JWT 验证和请求校验

测试重点放在了最高价值的地方（业务逻辑 + schema 校验），而不是追求全覆盖。

## 二、过程中的修正与 Learnings

### 1. 追加需求的处理

功能 9（提醒）和功能 10-11（Auth + Landing Page）都是项目中途追加的需求。我们没有匆忙实现，而是：
1. 先更新需求文档（bootcamp-assignment.md 追加附录）
2. 再做完整设计（PRD、API Design、Database Design 同步更新）
3. 新增设计决策（DD-014、DD-015）
4. 最后才实现

**Learning**：即使是 demo 项目，追加需求也应该走设计流程。这避免了"先写代码再补文档"导致的文档与实现漂移。

### 2. Review Lessons 的积累

`review-lessons.md` 记录了开发过程中的 UI 和代码 review 发现（如 UI-001 图标统一用 Lucide、UI-002 底部导航用 backdrop-blur）。虽然目前条目还不多，但这种"边做边记"的习惯为后续迭代提供了可复用的经验。

### 3. 单进程部署策略

最终采用了 server 同时服务 API 和前端静态文件的单进程方案（`pnpm build && pnpm start`），而不是前后端分别部署。这对 demo 项目来说是正确的简化——一条命令启动，评审者体验最佳。

### 4. 预置 Demo 账号

在 Auth 设计中加入 `demo/demo1234` 预置账号是一个好决定。评审者不需要注册就能体验完整功能，同时"现场注册"的路径也保留了展示完整 auth 流程的能力。

## 三、后续可以做得更好的方向

### 1. 前端状态管理

当前前端数据获取和状态管理相对简单（直接 fetch + useState）。如果项目继续迭代，可以考虑：
- 引入 React Query / TanStack Query 做数据缓存和乐观更新
- 将打卡操作的乐观更新做得更流畅（先更新 UI，再等 API 响应）

### 2. E2E 测试

目前测试覆盖了 schema → service → route 三层，但缺少端到端测试（如 Playwright / Cypress）。关键用户旅程（注册 → 创建习惯 → 打卡 → 查看 streak）如果有 E2E 保障会更完整。

### 3. 错误处理的用户体验

后端有统一错误处理，但前端对网络错误、token 过期等场景的用户提示可以更友好。例如：
- Token 过期时自动跳转登录页并保留当前路径
- 网络断开时显示离线提示而非空白页

### 4. 性能指标监控

代码审查确认了 demo 规模下性能可接受，但没有实际的性能指标采集。如果要正式上线，应该加入：
- 首屏加载时间（LCP）
- API 响应时间 P95
- 前端 bundle size 监控

### 5. CI/CD Pipeline

项目缺少自动化流水线。理想状态下应该有：
- PR 时自动运行 lint + test
- 主分支合并后自动构建和部署
- 测试覆盖率报告

### 6. 文档的可视化补充

设计文档以文字为主，wireframes 是 ASCII art。如果时间充裕，可以补充：
- Figma 或 Excalidraw 的可视化原型
- 数据库 ER 图
- API 交互时序图

### 7. 无障碍（Accessibility）

shadcn/ui + Radix 提供了基础的 a11y 支持（键盘导航、ARIA），但没有做过专门的无障碍测试。可以补充：
- 屏幕阅读器测试
- 颜色对比度检查
- 键盘完整操作流验证

# Habit Tracker - 用户交互流程 (User Flows)

> 版本：v1.0 | 更新日期：2026-03-10

---

## 流程 1: 首次使用 (First-time User Flow)

**入口：** 用户首次打开应用

```
1. 用户打开应用
   → 系统检测到无习惯数据

2. 系统显示欢迎页
   → 简短介绍文案：「用科学的方法，养成更好的习惯」
   → 展示核心理念：习惯回路（提示→行动→奖赏）简图
   → [开始创建第一个习惯] 按钮

3. 用户点击按钮
   → 进入创建习惯流程（流程 2）
   → 注意：首次创建时"习惯叠加"选项不可用（无已有习惯）

4. 创建完成
   → 返回主页，展示第一个习惯
   → 显示新手引导提示：「点击左侧的圆圈完成今日打卡」
```

**边界情况：**
- 用户关闭欢迎页但不创建习惯 → 主页展示空状态，保留"创建习惯"入口

## 流程 2: 创建习惯 (Create Habit Flow)

**入口：** 主页右下角 [+] 按钮 / 欢迎页引导

```
步骤 1: 基础信息填写
┌─────────────────────────────────────────┐
│ 1. 用户点击 [+创建习惯]                  │
│    → 系统打开创建表单（页面或底部弹窗）    │
│                                         │
│ 2. 用户填写习惯名称（必填）               │
│    → 实时字数提示 (0/50)                 │
│    → 为空时提交按钮禁用                   │
│                                         │
│ 3. 用户填写描述（可选）                   │
│    → 占位符提示："简单描述这个习惯..."     │
│                                         │
│ 4. 用户选择开始日期                      │
│    → 默认今天，日期选择器不可选过去日期    │
│                                         │
│ 5. 频次显示为"每日"（不可修改，灰色标签） │
│                                         │
│ 6. 用户点击 [下一步]                     │
│    → 进入 Cue 设置步骤                   │
└─────────────────────────────────────────┘

步骤 2: Cue 设置（必选其一）
┌─────────────────────────────────────────┐
│ 7. 系统展示两种 Cue 模式，引导文案：      │
│    「设置一个触发场景，帮助大脑建立习惯回路」│
│                                         │
│ 方式 A — 触发提示 (Trigger Cue):         │
│ 8a. 系统展示预设场景列表：                │
│     🌅 起床后 / 🪥 刷牙后 / 🍽️ 早餐后   │
│     🍽️ 午餐后 / 🍽️ 晚餐后 / 🏠 到家后   │
│     🌙 睡觉前                            │
│ 9a. 用户选择一个预设或点击 [自定义]       │
│     → 自定义：输入框，如"跑步后"         │
│ 10a. 系统生成预览：                      │
│     「如果 [午餐后]，那么 [阅读20分钟]」  │
│                                         │
│ 方式 B — 习惯叠加 (Habit Stacking):      │
│ 8b. 系统展示已有习惯列表                  │
│     → 无已有习惯时此选项置灰并提示        │
│ 9b. 用户选择一个已有习惯                  │
│ 10b. 系统生成预览：                      │
│     「在 [运动30分钟] 之后，我将 [阅读]」 │
│                                         │
│ 11. 用户点击 [创建]                      │
│     → 至少选择了一种 Cue：创建成功        │
│     → 未选择任何 Cue：提示"请选择至少一种  │
│       触发方式"，按钮保持禁用             │
└─────────────────────────────────────────┘

步骤 3: 创建完成
┌─────────────────────────────────────────┐
│ 12. 系统保存习惯数据                     │
│     → 显示成功 toast：「新习惯已创建！」   │
│     → 返回主页，新习惯出现在待完成列表顶部 │
│     → 进度条分母 +1                      │
└─────────────────────────────────────────┘
```

**错误状态：**
- 习惯名称为空 → 输入框下方红色提示"请输入习惯名称"
- 达到 5 个活跃习惯上限 → [+] 按钮变灰，点击后弹出科学提示（见 PRD §3.1）
- 网络错误 → "保存失败，请重试"提示 + 保留表单数据

## 流程 3: 每日打卡 (Daily Check-in Flow)

**入口：** 用户打开应用主页

```
1. 用户打开应用
   → 系统加载今日习惯列表
   → 顶部显示进度条：▓▓░░░ 1/5 已完成
   → 进度条下方展示 AI 每日鼓励语（见流程 3b）

2. 用户看到待完成列表
   → 未完成习惯在上方，按 Cue 时间或创建顺序排列
   → 已完成习惯在下方"已完成"分隔线之后

3. 用户点击某习惯的复选框
   → 复选框立即填充（乐观更新，< 200ms）
   → 该习惯卡片平滑动画移至"已完成"区
   → 进度条实时更新

4. 系统判断反馈类型：
   ┌──────────────────────────────────┐
   │ 普通打卡？                       │
   │   → 显示朋友式 toast（2-3秒）    │
   │   → 如「又搞定一个！」           │
   │                                  │
   │ 是今日最后一个？                  │
   │   → 触发全部完成庆祝（见流程 3a） │
   │                                  │
   │ 是里程碑节点？(7/21/30/66/100天) │
   │   → 触发里程碑回顾（见流程 5）    │
   └──────────────────────────────────┘

5. 系统后台更新数据
   → 创建 CheckIn 记录（date = 今天）
   → 重新计算 Streak
   → 检查里程碑触发条件
```

### 流程 3a: 全部完成庆祝

```
1. 最后一个习惯打卡完成
   → 进度条填满，变为完成色
   → 触发灵动的庆祝动效

2. 显示温暖文案
   → 如「今天全部达成！你对自己的承诺从未落空。」
   → 文案展示 3-4 秒后自动淡出

3. 主页进入"今日全部完成"状态
   → 所有习惯在"已完成"区
   → 进度条保持完成状态
   → AI 鼓励语刷新为庆祝风格（如需要）
```

### 流程 3b: AI 每日鼓励展示

```
触发时机：用户打开首页，系统加载 /api/today 数据时

1. 系统根据当前状态生成鼓励语
   → 调用 Claude Haiku API（服务端），传入：
     - 各习惯的 streak 天数
     - 今日打卡进度
     - 是否有 streak 断裂
     - 习惯名称
   → 每个习惯每天只生成一次，后续使用缓存

2. 鼓励语展示在进度条下方、习惯列表上方
   → 1-2 句话 + 颜文字/表情符号
   → 风格随状态自然切换：
     ┌──────────────────────────────────┐
     │ 日常坚持 → 轻松朋友式            │
     │ Streak 断裂 → 温暖安慰式         │
     │ 接近里程碑 → 兴奋冲刺式          │
     │ 全部完成 → 庆祝式                │
     │ 第 1 天 → 启程鼓励式             │
     └──────────────────────────────────┘

3. Fallback 处理
   → AI API 不可用或超时 → 从预置静态文案库随机选取
   → 用户体验无差异

4. 加载中状态
   → 显示骨架占位（shimmer），避免布局抖动
```

## 流程 4: 补卡 (Retroactive Check-in Flow)

**入口：** 习惯卡片上的"昨日未打卡"标签

```
前置条件检查：
┌──────────────────────────────────────┐
│ 系统在每次加载列表时检查每个习惯：    │
│                                      │
│ 昨日未打卡？                         │
│   AND streak > 0（或开始日期=昨天）？ │
│   AND 本月补卡次数 < 3？             │
│   → 显示补卡标签                     │
│                                      │
│ 否则：                               │
│   → 不显示补卡入口                   │
└──────────────────────────────────────┘

1. 用户看到习惯卡片上的标签：「昨日未打卡，补卡？」
   → 标签使用区别于常规状态的颜色（如橙色）

2. 用户点击补卡标签
   → 系统弹出确认对话框：
     标题：「补打昨日打卡」
     内容：「为"[习惯名称]"补打 [昨日日期] 的卡？」
     信息：「本月剩余补卡：X/3」
     按钮：[取消] [确认补卡]

3. 用户点击 [确认补卡]
   → 系统创建 CheckIn 记录（date = 昨天，isRetroactive = true）
   → Streak 正常延续计算
   → 补卡标签消失
   → 显示 toast：「补卡成功，streak 继续！」
   → 本月补卡计数 +1

4. 补卡配额用尽时：
   → 补卡标签变为置灰样式
   → 点击后提示：「本月补卡次数已用完，下月将重置」
```

**边界情况：**
- 用户今天已经打过卡但昨天没打 → 仍可补打昨天（补卡与今日打卡独立）
- 跨月补卡（今天是 1 号，补昨天即上月最后一天）→ 使用本月（新月）的补卡配额
- 习惯开始日期是今天 → 不显示补卡（昨天不在习惯范围内）

## 流程 5: 里程碑回顾 (Milestone Review Flow)

**入口：** 打卡完成后自动触发

```
触发条件检查：
┌──────────────────────────────────────┐
│ 打卡完成后，系统检查该习惯的累计      │
│ 打卡天数（非连续，而是总打卡天数）：   │
│                                      │
│ 总天数 ∈ {7, 21, 30, 66, 100}？     │
│   AND 该里程碑未被触发过？            │
│   → 触发里程碑回顾                   │
└──────────────────────────────────────┘

1. 打卡动画结束后（约 0.5 秒延迟）
   → 系统弹出里程碑回顾卡片（模态覆盖层）

2. 回顾卡片内容：
   ┌──────────────────────────────┐
   │ 🎯 里程碑徽章                │
   │ 如：「🔥 7 天成就」          │
   │                              │
   │ 📊 数据区域                  │
   │ · 总打卡天数：7 天           │
   │ · 完成率：100%               │
   │ · 最长连续天数：7 天          │
   │                              │
   │ 💬 激励文案区域              │
   │ 「连续 7 天！你的大脑正在     │
   │   建立新的神经通路，坚持下去。」│
   │                              │
   │        [知道了 ✓]            │
   └──────────────────────────────┘

3. 用户点击 [知道了]
   → 关闭回顾卡片
   → 系统记录 MilestoneEvent（dismissed = true）
   → 返回主页正常状态

4. 用户点击遮罩层外部
   → 同样关闭（等同于点击"知道了"）
```

**各里程碑文案风格：**
| 天数 | 风格 | 关键词 |
|------|------|--------|
| 7天 | 导师式 | 神经通路、坚持 |
| 21天 | 深度科学 | 习惯形成起点、神经层面变化 |
| 30天 | 数据导向 | 月度完整报告、趋势 |
| 66天 | 诗意庆祝 | "自然而然"、科学平均值 |
| 100天 | 成就庆祝 | 百日里程碑、深度肯定 |

## 流程 6: 编辑习惯 (Edit Habit Flow)

**入口：** 习惯卡片上的操作菜单 → [编辑]

```
1. 用户在习惯卡片上点击操作菜单（⋯ 或 ⋮）
   → 弹出下拉菜单：[编辑] [删除]

2. 用户点击 [编辑]
   → 打开编辑表单页面，预填当前数据：
     - 习惯名称（当前值）
     - 描述（当前值）
     - 开始日期（当前值）
     - 当前 Cue 类型和值（高亮显示）

3. 用户修改所需字段

4. 用户修改 Cue 设置时
   → 显示引导提示：「调整触发场景有助于重新找到适合你的节奏」
   → Cue 切换逻辑同创建流程
   → 至少保留一种 Cue 方式

5. 用户点击 [保存]
   → 系统验证字段（同创建时的规则）
   → 保存成功 → toast「习惯已更新」→ 返回列表
   → 打卡记录和 Streak 不受影响

6. 用户点击 [取消] 或返回
   → 放弃修改，返回列表
   → 如有未保存修改，提示「放弃未保存的修改？」
```

## 流程 7: 删除习惯 (Delete Habit Flow)

**入口：** 习惯卡片上的操作菜单 → [删除]

```
1. 用户在操作菜单中点击 [删除]
   → 弹出确认对话框

2. 确认对话框内容：
   ┌───────────────────────────────────┐
   │ ⚠️ 删除习惯                       │
   │                                   │
   │ 确定要删除"阅读 20 分钟"吗？      │
   │ 所有打卡记录将一并删除，且无法恢复。│
   │                                   │
   │    [取消]       [确认删除]         │
   │                  (红色)            │
   └───────────────────────────────────┘

3a. 用户点击 [确认删除]
   → 习惯及所有关联数据删除
   → 列表刷新，习惯消失
   → 进度条更新（分母 -1）
   → toast「习惯已删除」
   → 活跃习惯数 -1

3b. 用户点击 [取消]
   → 关闭对话框，无任何变化
```

## 流程 8: Streak 断裂处理 (Streak Break Flow)

**入口：** 用户打开应用时自动检测

```
系统检测逻辑：
┌──────────────────────────────────────┐
│ 应用加载时，对每个习惯检查：          │
│                                      │
│ 昨日未打卡？                         │
│   AND 前天已打卡（即 streak 刚断裂）？│
│   AND 该次断裂的安慰消息未展示过？     │
│   → 显示断裂安慰                     │
└──────────────────────────────────────┘

1. 用户打开应用
   → 系统检测到某习惯 streak 断裂

2. 在该习惯卡片上方或内部展示安慰消息：
   ┌───────────────────────────────────┐
   │ 💙 「没关系。研究表明，偶尔的中断  │
   │ 并不会前功尽弃——你之前建立的神经   │
   │ 通路仍然在那里。重新开始，比第一次  │
   │ 容易得多。」                       │
   │                           [关闭 ×] │
   └───────────────────────────────────┘

3. 同时判断是否可补卡：
   → 可补卡：安慰消息下方显示补卡入口「补打昨日的卡？」
   → 不可补卡（配额已尽）：仅显示安慰消息

4. 用户可选操作：
   a. 点击补卡 → 进入补卡流程（流程 4）
   b. 点击关闭 → 安慰消息消失，标记为已展示
   c. 直接打今日的卡 → 安慰消息在打卡后自动消失

5. 系统记录：该次断裂的安慰消息已展示
   → 后续进入应用不再重复此消息
```

## 流程 9: 习惯提醒通知 (Habit Reminder Flow) — 追加需求

### 流程 9a: 设置提醒时间

**入口：** 创建习惯步骤一 / 编辑习惯表单

```
1. 用户在创建/编辑习惯表单中看到"提醒时间"字段
   → 默认为空（不提醒）

2. 用户点击提醒时间字段
   → 弹出时间选择器（HH:mm）
   → 提供快捷预设按钮：[早上 8:00] [中午 12:00] [晚上 20:00]
   → 用户可手动选择任意时间

3. 用户选择时间后
   → 字段显示选中时间，如"每天 08:00 提醒"
   → 旁边出现 [×] 清除按钮

4. 首次设置提醒时间时检查通知权限：
   ┌──────────────────────────────────────┐
   │ 权限状态 = "default"（未请求过）？    │
   │   → 调用 Notification.requestPermission()│
   │   → 用户允许 → 正常继续               │
   │   → 用户拒绝 → 显示提示文案并继续保存  │
   │                                      │
   │ 权限状态 = "denied"（已拒绝）？       │
   │   → 显示提示：「通知权限已被禁用，     │
   │     请在浏览器设置中开启」             │
   │   → 提醒时间仍可保存                  │
   │                                      │
   │ 权限状态 = "granted"（已允许）？      │
   │   → 无需额外操作                      │
   └──────────────────────────────────────┘

5. 用户提交表单（创建/保存）
   → reminderTime 随其他字段一起保存
```

### 流程 9b: 接收提醒通知

**入口：** 系统自动触发（到达设定时间）

```
前台运行时：
┌──────────────────────────────────────┐
│ 客户端每分钟检查一次：               │
│                                      │
│ 当前时间 = 某习惯的 reminderTime？    │
│   AND 今日该习惯未打卡？              │
│   AND 通知权限 = "granted"？         │
│   AND 今天未曾为该习惯发送过提醒？     │
│   → 发送浏览器通知                   │
│   → 记录已发送（防止重复）            │
└──────────────────────────────────────┘

后台/非活跃标签时：
┌──────────────────────────────────────┐
│ Service Worker 接管定时检查：          │
│   → 同样的触发逻辑                   │
│   → 通过 Service Worker 发送通知      │
└──────────────────────────────────────┘

1. 系统发送通知
   → 标题：⏰ 习惯打卡提醒
   → 正文：「阅读 20 分钟」— 现在是你的习惯时间！

2. 用户点击通知
   → 聚焦到应用标签页
   → 对应习惯卡片高亮或滚动到可见位置

3. 用户忽略通知
   → 通知自动消失（跟随系统设置）
   → 今日不再重复提醒该习惯
```

### 流程 9c: 页面 Badge 更新（Nice-to-have）

```
1. 应用加载后，计算未完成习惯数量
   → 未完成数 > 0：document.title = "(3) Habit Tracker"
   → 全部完成或无习惯：document.title = "Habit Tracker"

2. 每次打卡后实时更新
   → 重新计算未完成数量
   → 更新 document.title
```
